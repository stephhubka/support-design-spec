Section 02

Case queue & triage

The Inbox is the primary interface for the Support v2 case queue â€” where agents triage work by scanning, filtering, and selecting cases to resolve. This section defines the queue experience across both board and list-based views, including the shared controls, case preview model, and key interactions that occur before entering a conversation.

Supported views

The case queue can be represented in multiple formats depending on workflow needs.

Board view

Best for status-based triage and workflow movement. Cases are grouped by status to support quick prioritization, but Phase 1 does not include drag-and-drop between columns â€” status changes occur through explicit update actions to ensure clarity and auditability.

List view

Best for high-volume scanning, sorting, and bulk management. List view supports consistent queue indicators and actions while providing a more compact, sortable representation of active work.

Case preview information

Before opening a case, agents should have enough context to prioritize work directly from the queue. Case previews include:

  • Customer name + avatar
  • Message preview
  • Status and priority indicators
  • Channel badge
  • Timestamp and urgency cues

Shared controls across views

These controls remain consistent regardless of queue representation:

Search + filter bar
Search is available across the queue to help agents locate cases by customer or conversation context. Support reps can switch between board and list views while preserving the same underlying case set and applied filters.

Key triage interactions

The primary actions supported within the queue include:

Select and open a case

Support reps can open a case directly from the card or row, at which point unread indicators clear once messages are viewed.

Bulk actions

Bulk actions are supported in Phase 1 for a focused set of high-value workflows:

  • Update case status
  • Apply priority

Additional bulk updates (tags, categories, routing) are intentionally deferred until later releases.

AI-assisted guidance (Phase 1)

In Phase 1, the Conversation Workspace integrates AI as a read-only support layer. AI helps agents understand context, surface relevant information, and draft guidance â€” but does not take action on behalf of the support rep.

AI suggestions panel

Phase 1 capabilities include:

  • Intent recognition
  • Contextual suggestions
  • Always-updated conversation summaries
  • Retrieved supporting data displayed for the support rep

Planned enhancements (Phase 2+)

Write-enabled AI workflows may be introduced in future phases, such as:

  • Drafting and applying suggested replies
  • Executing approved case updates (status, assignment)
  • Triggering workflow actions with explicit support rep confirmation

All write actions will require clear guardrails, transparency, and human approval before execution.

Phase 1 Scope: AI operations are read-only. The agent may retrieve and display data, but cannot execute actions or updates on behalf of the support rep. Write-enabled workflows are planned for Phase 2+.

Timezone Dropdown Used for business hours, SLA calculations, and timestamps Required, defaults to institution's primary timezone

Timezone behavior

  • Changing the timezone triggers a confirmation dialog showing the from / to change and warning that all active SLA deadlines will be recalculated. This cannot be undone.
  • Timestamps displayed to reps adjust to the workspace timezone. Customer-facing timestamps use the customer's local timezone where available.
  • The timezone also governs when "Outside business hours" auto-response triggers fire (see Section 08: AI Configuration).

Business hours

Set when the support team is available. Customers will see these hours when starting a conversation.

Behavior

  • Time drop-downs use 30-minute increments (8:00 AM, 8:30 AM, 9:00 AM, etc.) in 12-hour format.
  • Validation: End time must be after start time. If a user selects an invalid range, the end time dropdown shows an error.
  • Toggling a day off sets it to "Closed" and clears any previously set times. Toggling it back on restores the last used times for that day (or defaults to 8:00 AM â€“ 6:00 PM).
  • Split shifts are not supported in v1. Each day has a single continuous block.
  • Changes take effect immediately on save; no scheduled rollout. Active SLA clocks recalculate based on new hours.

Dependencies

Business hours are referenced by:

  • SLA clock (section 12): pauses outside these hours when "Business hours only" is enabled on a policy.
  • "Outside business hours" auto-responses (section 08): triggers when a customer message arrives outside these hours.
  • Queue displays: agent see a "Currently outside business hours" banner when applicable.

Holidays and closures

Define dates when your regular hours don't apply. Your holiday message will override standard messaging on these dates. New holidays default to: single day, non-recurring.

Section 11

Rules

Automatically assign cases and set response time targets.

Routing rules behavior

This section covers behavior that isn't visible in the designs.

Rule Evaluation

  • Rules are evaluated top to bottom in the order they appear in the list. First match wins â€” once a case matches a rule, no further rules are evaluated.
  • Drag to reorder rules. This is the primary mechanism for controlling priority.
  • If no rules match, the case goes to the default queue (unassigned).

Toggle behavior

  • Disabling a rule skips it during evaluation â€” the case proceeds to the next rule in the list.
  • Disabling does not unassign cases already routed by that rule.
  • Re-enabling a rule applies it to new incoming cases only, not retroactively.
  • Disabled row dim to 45% opacity with a "Disabled" label beside the rule name.

Conditions

Condition Operators Example
Category equals, not equals category = Dispute / Fraud
Customer segment equals, not equals customer segment = Business
Channel equals, not equals channel = Email
Priority equals, greater than, less than priority > High
Keyword contains message contains "wire transfer"
  • Rules support single conditions in v1. Compound conditions (AND / OR) are a v2 consideration.
  • Category and priority values come from the workspace's configured taxonomy. If a category is deleted, rules referencing it are auto-disabled with a warning badge.

SLAs behavior

Default SLA policies

The four default policies cannot be deleted, only edited. They map to priority levels:

Priority First response target Resolution target Escalation
Urgent 1 hour 4 hours Auto-escalate to team lead after breach
High 4 hours 12 hours Auto-escalate to team lead after breach
Medium 24 hours 48 hours Notify assignee at approaching threshold
Low 48 hours 72 hours Notify assignee at approaching threshold

Clock Behavior

  • SLA clock starts when a case is created (not when it's assigned).
  • Clock runs on business hours by default (pulled from Settings â†’ General â†’ Business hours). Weekends and holidays are excluded.
  • Option to override to calendar hours per SLA policy.
  • Clock pauses when case status is set to "Waiting on customer." Resumes when the customer replies or a support rep manually resumes.

Edit Drawer

Custom SLA policies

Admins can create additional SLA policies via "+ Add SLA" for cases that don't fit the four policy tiers (e.g., VIP customers, regulatory inquiries). Custom policies require:

  • A unique name
  • At least one target (first response or resolution)
  • An assignment condition â€” which cases this policy applies to (condition builder, same format as routing rules)

Custom policies are evaluated after priority-based defaults. If a case matches both a default and custom policy, the stricter target (shorter time) applies.

Empty state

Not applicable. The four default policies are always present.

Section 12

AI

Configure how AI helps teams respond to customers. The Settings â†’ AI tab contains two sections: AI Assistant (real-time agent tools) and auto-responses (trigger-based customer messages).

AI Assistant - Behavior

This section covers behaviors that aren't visible in the designs.

Toggle behavior

  • Changes take effect on the next case load or page refresh. Mid-conversation, the feature stays visible until the agent navigates away.
  • AI Sidekick â†’ Suggested Replies dependency: Toggling off Sidekick also disables Suggested Replies.
  • Auto-categorization: Toggling off does not remove existing categories. Only stops new auto-assignment.

Auto-categorization confidence

  • Fixed internal threshold at 80%. Below threshold â†’ case is left "Uncategorized" for manual assignment.
  • Agent corrections are logged as implicit feedback. Surface monthly accuracy summary in this settings tab.
  • V2 consideration: Replace fixed threshold with plain-language options if accuracy varies across institutions.
  • Workspace admins only. Changes logged in audit trail.

Trigger priority

When multiple triggers match, only one fires. Priority order:

  1. Holiday closure (highest)
  2. Out of hours greeting
  3. Custom triggers (in list order; drag to reorder)
  4. Initial acknowledgement (lowest; fires only if nothing else did)

Example: New case on Christmas at 9 PM â†’ Holiday closure fires. Initial acknowledgement does not.

Rate limiting

Default 24-hour cooldown per auto-response per customer. Same trigger for same customer within the window is suppressed. Configurable per response: 1hr, 4hr, 12hr, 24hr (default), 48hr, No limit.
